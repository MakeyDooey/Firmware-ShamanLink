Project Description:
I want to program an LPC55S69 using MCUXpresso IDE to serve as a high-performance programmer and bus orchestrator for STM32H7 and STM32G0 MCUs. I have a custom PCB and will use MCULink hardware to initially flash the LPC55S69 via its SWD pins. The goal is for a user to connect a computer to the PCB's USB-C port and interact with a FreeRTOS+CLI (instantiated over a USB CDC VCOM interface) to execute commands. These commands include flashing a target board via the SWD protocol using designated pins, sending/receiving messages over a half-duplex multi-drop RS485 bus where the LPC55S69, H7, and G0 all reside, and manually toggle certain on-board user LEDs on or off.

The design must leverage the LPC55S69’s dual-core architecture: Core 0 (master) will manage the FreeRTOS tasks, USB stack, and CLI parsing, while Core 1 (worker) will handle timing-critical SWD bit-banging and low-level RS485 direction control (DE/RE) to ensure high-speed, reliable communication. Inter-core communication will be handled via OpenAMP (RPMsg-Lite) to create a seamless data pipeline. When flashing a target, no files should be stored on the NXP; instead, data should be streamed from the host computer, passed through the OpenAMP buffers, and written to the target in real-time. Finally, the system will use onboard LEDs to provide visual telemetry, such as indicating an active USB connection, bus activity, or specific flashing errors.

(CDC USB Wikipedia) 
Abstract Control Model (ACM): Protocol defined on the CDC specification that can be used to implement a general-purpose serial UART. Commonly implemented in microcontrollers to provide a serial console for configuring and debugging embedded system. 

(Virtual COM port Wikipedia) 
A software representation of a serial port that either does not connect to a real serial port, or adds functionality to a real serial port through software extension.

Feature Descriptions
 - Note: the MCU-Link can be back-powered by a target if the MCU-Link itself is not powered via USB. For this reason it is recommended that the power be applied to the MCU-Link before the target.
 - The MCU-Link Base Probe cannot power a target system, so uses a sensing circuit to detect the target supply voltage and set up level shifter voltages accordingly. 
 - If issues are seen with the target system supply being affected by the MCU-Link being connected then R16 may be removed and SJ1 changed to connect to position 1-2. This will fix the level shifters at the voltage level seen at pin 1 of the SWD connector, and require that the target supply can support the VCCB input requirements of the level shifter devices.

PRD: Professional USB-CLI control system

1. Hardware Specification & Pin Mapping

The system targets the LPC55S69JBD64 (U1). Control of the RGB LED is achieved via three dedicated GPIO pins on Port 1. Because the hardware is wired in an Active-Low configuration (Common Anode), the GPIO must sink current (Logic 0) to illuminate the LED.

Signal	GPIO	Schematic Pin	Active Level	SDK Constant
Red	PIO1_1	39		Low (0=ON)	BOARD_LED_RED_GPIO_PIN
Blue	PIO1_2	41		Low (0=ON)	BOARD_LED_BLUE_GPIO_PIN
Green	PIO1_3	42		Low (0=ON)	BOARD_LED_GREEN_GPIO_PIN

1.1 Clocking & USB PHY Configuration

- Reference Clock: External 32MHz Crystal (Y1) on Pins 19/20.
- USB Controller: USB1 (High-Speed IP).
- PHY Requirements: The USB1 PLL must be engaged to provide the 480MHz PHY clock.
- Clock Tree: Main Clock at 150MHz (PLL0); USB1 High Speed Clock sourced from the USB PLL.
- VBUS Sensing: Mandatory sensing on Pin 26 (USB1_VBUS). USB_DeviceCdcVcomInit must use physical VBUS detection to support hot-plugging.

2. Software Architecture (MCUXpresso SDK Based)

The firmware is built upon the middleware.usb.device and middleware.freertos-kernel components.

2.1 Baseline SDK Examples

- dev_cdc_vcom_freertos: Handles USB stack, endpoint management, and CDC enumeration.
- USB String Descriptors: The usb_device_descriptor.c must be modified to reflect the following identity:
	- Product String: "ShamanLink"
	- Manufacturer String: "MakeyDooey"

2.2 Task Priority & Scheduling

Task Name	Priority	Stack Size	Responsibility
USB_Device_Task	4 (High)	2048		USB stack maintenance and endpoint ISR handling.
CLI_Task	3		4096		String parsing, command execution, and LED GPIO control.
Idle_Task	0		-		System sleep / low-power entry.

2.3 Buffer Strategy

- Mechanism: StreamBufferHandle_t (Thread-safe FIFO).
- RX Buffer (512B): USB ISR pushes raw bytes here via xStreamBufferSendFromISR.
- TX Buffer (1024B): CLI task pushes response strings; USB task pulls and sends via USB_DeviceCdcAcmSend.

3. CLI Command Interface & Logic

The CLI shall implement a VT100-compatible interface. Logic Note: Since LEDs are Active-Low, the software logic is inverted: Logic 0 = ON, Logic 1 = OFF.

3.1 Color Logic Matrix (Active-Low)

Command String 	Red (1_1)	Blue (1_2)	Green (1_3)
rgb red 	0		1		1
rgb green	1		1		0
rgb blue	1		0		1
rgb yellow	0		1		0
rgb magenta	0		0		1
rgb cyan	1		0		0
rgb white	0		0		0
rgb off		1		1		1

3.2 Error Handling & State

- Invalid Input: Returns "Error: Unknown color. Type 'help' for valid options."
- Boot State: All GPIOs (39, 41, 42) must be initialized to Logic 1 in BOARD_InitPins().
	- Warning: Initializing to Logic 0 would cause the LED to glow white during the boot/enumeration phase.

4. Success Criteria
- Enumeration: Device appears as "LPC55S69 USB High-Speed VCOM" in Host Device Manager.
- Logic Verification: Scope probe confirms Logic High (3.3V) on pins when rgb off is sent.
- Latency: Physical LED state must update within 10ms of the carriage return character (\r) reception.
- Stability: High-Speed PHY must sustain rapid-fire CLI input without buffer overflow.
